# Body preservation: stop nuking LLM-authored content

> **Status: IMPLEMENTED.** All changes below have been made. See [`body-preservation-invariants.md`](body-preservation-invariants.md) for the tested invariants proving the fix works.

## The problem

`figmaclaw enrich` (and `pull`) unconditionally call `write_page()` which calls `render_page()`, regenerating the **entire** .md file from scratch — frontmatter AND body. This destroys all LLM-authored prose: page summaries, section intros, filled description tables, and Mermaid charts.

The body is expensive to produce (Figma screenshots + LLM inference + human review). Losing it on every sync is unacceptable.

## The design contract (already documented, not enforced)

- **Frontmatter** = machine-readable source of truth. Code reads/writes this.
- **Body** = LLM/human prose. **Only written by LLM/human. Never regenerated by code.**
- `render_page()` writes a skeleton body for **new pages only**.

The contract is stated in docstrings throughout `figma_render.py` and `pull_logic.py`, but the code violates it: `enrich` and `pull` call `render_page()` on existing files, nuking the body every time.

## Renaming for correct semantics

| Current name | Actual behavior | New name |
|---|---|---|
| `enrich` (CLI command) | Syncs frontmatter structure from Figma REST API | **`sync`** |
| `render_page()` | Writes initial skeleton .md with placeholders | **`scaffold_page()`** |
| `write_page()` | Calls `render_page()` and writes to disk | **`write_new_page()`** |

`enrich` implies enriching content — it doesn't, it syncs structure. `render_page` implies rendering — it only writes an initial scaffold. Names should match what the code does.

## Changes required

### 1. `figmaclaw/commands/enrich.py` -> rename to `sync.py`

When the file **already exists**: update ONLY the frontmatter using `python-frontmatter` library (already a dependency — `import frontmatter` in `figma_parse.py`). Never call `write_new_page()`/`scaffold_page()` on existing files.

When the file **doesn't exist**: write skeleton via `write_new_page()` as normal.

### 2. `figmaclaw/commands/set_frames.py`

Frontmatter only — body is never touched by code. The LLM uses `replace-body` to update body prose.

### 3. `figmaclaw/pull_logic.py`

Same fix as `sync.py`: for existing files, update only frontmatter. For new files, write skeleton.

### 4. `figmaclaw/figma_render.py`

Rename `render_page()` to `scaffold_page()`. Rename `write_page()` to `write_new_page()` (in `pull_logic.py`).

Add a **print/return mode** to `scaffold_page()` so it can output the scaffold to stdout without writing to disk. This is used by the LLM as a structural hint when rewriting the body after significant page changes.

## `scaffold_page()` design

### Two modes

1. **Write mode** (new file only) — writes the skeleton .md to disk. Only called when the file doesn't exist yet.
2. **Print/return mode** — returns the scaffold as a string (or prints to stdout) without touching any file. Used by the LLM as a template when the page structure changed significantly.

### Scaffold format with LLM placeholders

The scaffold should contain explicit placeholders telling the LLM exactly what to fill in:

```markdown
---
file_key: ...
frames: {... merged from Figma ...}
flows: [...]
---

# {file} / {page}

[Open in Figma](...)

<!-- LLM: Write a 2-3 sentence page summary describing what this page covers -->

## {Section Name} (`{node_id}`)

<!-- LLM: Write a 1-sentence section intro if the section has a distinct theme -->

| Screen | Node ID | Description |
|--------|---------|-------------|
| {frame_name} | `{node_id}` | <!-- LLM: <=20 words, what it shows + what makes it distinct --> |
| ... | ... | ... |

## Screen Flow

<!-- LLM: Generate Mermaid flowchart from the flows in frontmatter, or omit if no flows -->
```

### The LLM's three inputs when updating a page

1. **Existing body** — verbatim, the current prose to preserve/adapt
2. **Updated frontmatter** — from `sync` command, shows what frames/flows exist now (new frames, removed frames, changed structure)
3. **Scaffold (optional)** — the template showing expected output structure with placeholders, so the LLM knows exactly what sections to write and where

The scaffold is a contract for the LLM, not a destructive rewrite.

### The LLM must ALWAYS see the existing body

The LLM should **never** write a body blind. It must always receive the current body verbatim so it can preserve prose, adapt existing descriptions, and only change what actually needs changing. Rewriting from scratch loses voice, context, and human edits.

The figmaclaw tooling can provide this trivially — the existing .md file already contains the body. `python-frontmatter` separates it cleanly:

```python
post = frontmatter.loads(md_path.read_text())
existing_body = post.content  # everything below the --- delimiters, untouched
```

The CLI could expose this explicitly, e.g. `figmaclaw sync --show-body` or `figmaclaw scaffold --with-existing-body`, outputting both the scaffold (what the new structure should look like) and the existing body (what the LLM should preserve/adapt) side by side. The skill prompt then hands both to the LLM with clear instructions: "Here is the current body. Here is the new structure. Rewrite the body to match the new structure while preserving existing prose."

## Library usage: no hand-parsing

All frontmatter manipulation must use the `python-frontmatter` library (`import frontmatter`), which is already a dependency. It cleanly separates frontmatter from body content:

```python
import frontmatter

post = frontmatter.loads(md_text)
# post.metadata = dict (the YAML frontmatter)
# post.content = str (the body, completely untouched)

# Update frontmatter fields
post.metadata["frames"] = new_frames_dict

# Write back — body is preserved exactly as-is
updated_md = frontmatter.dumps(post)
```

No `str.partition("---")`, no regex splitting, no hand-parsing. The library handles the `---` delimiters correctly in all edge cases.

**Note:** The existing `_build_frontmatter()` in `figma_render.py` uses custom YAML dumper classes (`_FlowDict`, `_FlowList`, `_FrontmatterDumper`) to get single-line flow-style for `frames` and `flows`. When using `python-frontmatter` to write back, ensure the same formatting is preserved — either by using a custom handler or by replacing the metadata string directly with `_build_frontmatter()` output.
